# 协议帧格式

协议层定义的帧结构和消息类型映射表。^[extracted]

## 帧结构

固定长度 **768 字节**：^[extracted]

```
| 帧头 | 序号 | 消息块 | CRC | 帧尾 |
| 1B  |  1B  |  N B   | 2B  | 1B  |

帧头: 0xA5
序号: SN (Sequence Number)，用于检测丢包重复乱序
消息块: 包含一个或多个逻辑消息，MessageBlock 可为 0 字节（心跳）
CRC:   2 字节校验
帧尾: 0xAD
```

固定帧长简化了 DMA 传输和缓冲区管理的复杂性。^[extracted]

## 帧类型

| 帧类型 | 用途 | 数据大小 |
|--------|------|----------|
| `FRAME_COMMON_DATA` | 常规车信号/BAP/诊断消息 | 变长 |
| `FRAME_FOTA` | OTA 升级数据 | 512+5 字节^[extracted] |
| `FRAME_LOG` | MCU 日志 | 变长 |

## 消息映射表

`rpc_message_type_infos[]` 定义了每种消息的详细信息：^[extracted]

- 消息类型枚举 → 2 字节协议 ID
- 对应的 GrpId（与 RpcDataLib 数据层对接）
- 发送/接收方向标志 (`SEND_NOTIFY`/`RECV_NOTIFY`/`SEND_REQUEST`)
- 自定义的 `set_parser` / `recv_parser` 回调函数

### 消息类别覆盖

- 电源管理
- 系统信息
- BLE 钥匙
- 诊断 UDS
- OTA 升级
- CAN 信号透传
- BAP 协议

## 序号机制

CNT 序号在传输层用于检测包异常：^[extracted]

- **丢包检测**：接收侧发现序号不连续则标记丢失
- **去重**：收到重复序号的包直接丢弃
- **乱序检测**：序号异常跳跃（如正常递增却出现大幅跳跃）视为故障

## 多消息复用

单帧可包含多个逻辑消息，MessageBlock 的组包顺序需要双方一致。^[extracted]

心跳场景下 MessageBlock 可为 0 字节，仅用于保活和确认链路畅通。^[inferred]
